# Debug investigations

Documented bugs and open questions from working with the OpenDAW SDK. Each file describes a specific behaviour, the mechanism (verified against current SDK source), and how to reproduce it using one of the unlisted debug demo pages.

## Conventions

- **One file per investigation.** Filename in kebab-case, descriptive, no date or number prefix. Sort by topic, not chronology.
- **Each note states the SDK version it was verified against.** Code citations (file:line) are point-in-time and decay — re-verify before quoting.
- **Focus on evidence and repro, not advocacy.** Document what's observed and how to reproduce it; label what's inferred from source-tracing as inferred. The goal is to make the symptom reproducible by anyone reading the note — diagnosis, recommendations, and contract questions are out of scope.
- **Repro pages are unlisted.** They live alongside the regular demos but are not added to `src/index.tsx` or `public/sitemap.xml`. The HTML carries `<meta name="robots" content="noindex, nofollow">`. They're reachable only by direct URL and are intentionally minimal — one button, one configuration, one thing to listen for.
- **Proven browser-probe techniques** (no SDK or page changes needed): patch
  `OfflineAudioContext.prototype.startRendering` to capture rendered buffers; intercept
  `MessagePort.prototype.postMessage` to log engine transport commands; pull `project`
  from a mounted component's React fiber. All three were used to close the seam and
  mode-swap investigations.
- **Engine-scoped verification:** the seam and voice-fade repro pages accept `?engine=wasm`
  to boot the WASM (Rust) engine (upstream scoped the #311/#312 fixes to it; #312 measured
  fixed on both engines, #311 on WASM only). Their
  offline scan routes through `OfflineEngineRenderer` with `variant: true` in that mode —
  `OfflineAudioContext` + `createEngine` throws with the WASM `EngineVariant` once any wasm
  engine booted on another context (`ensureReady` registers the processor only on the first
  context — see [wasm-ensure-ready-second-context.md](./wasm-ensure-ready-second-context.md)).
  The live WASM transport can take 20–30 s+ after `play()`
  before position advances on these pages; the offline scan does not depend on it.
- **Claims about runtime behaviour must be empirically verified, not inferred.** Tracing through SDK source produces plausible-looking explanations but can include unverified steps. If the doc states "X happens because Y," `Y` must be confirmed by logging the relevant value at runtime, stepping through with a debugger, or otherwise observing it directly. Citation of file:line for `Y` is not verification of `Y` — it's verification that the code at that location *exists*. When in doubt, present the symptom and repro without the mechanism, and let the maintainer diagnose.

## Index

- [time-pitch-start-position-pop.md](./time-pitch-start-position-pop.md) — Audible pop when starting playback inside a mid-file silent gap bracketed by audio, with `AudioTimeStretchBox` attached at `playbackRate = 1.0`. TimeStretch-specific (not in NoStretch, not on head-of-file silence). Mechanism identified: `TimeStretchSequencer.#handleTransientBoundary` starts the voice at the enclosing segment's onset rather than the playhead position, replaying pre-gap audio through the pitch-shift pipeline. Mode-swap "playhead reset" resolved — the page's own defensive `setPosition` call was the confounder. Repro: [`time-pitch-start-position-debug-demo.html`](../time-pitch-start-position-debug-demo.html).
- [splice-click-cross-file.md](./splice-click-cross-file.md) — Audible click at exact region boundaries when consecutive same-track regions reference different audio files. Open question: caller-managed fades or automatic voice handling? Repro: [`comp-lanes-debug-demo.html`](../comp-lanes-debug-demo.html).
- [fade-out-end-of-file-pop.md](./fade-out-end-of-file-pop.md) — Audible click at the end of a clip whose fade-out ends exactly at the audio file's end. Fixed upstream in core 0.0.145 (keep-guard added to `#updateOrCreatePitchVoice`); re-verified clean at SDK 0.0.154 / core 0.0.152. Page is now a regression check for this fix. Repro: [`fade-out-end-of-file-debug-demo.html`](../fade-out-end-of-file-debug-demo.html).
- [shared-source-double-process.md](./shared-source-double-process.md) — Sample-level discontinuity (≈ 2× the clean-sine `max |Δsample|`) 2 samples before adjacent same-track region seams. Mechanism closed: three stacked effects — `bpn` truncation forcing `seam−1` to zero, offline harness geometry artefact (both seam positions land at the same in-block offset), and off-boundary live geometry producing a 20 ms destructive crossfade at −9.7 dB. **Fixed in SDK 0.0.159 on the WASM engine** (openDAW#311 closed; all four cells scan at ratio 1.00 with `?engine=wasm`); TS engine still measures 1.87× at 0.0.159. Repro: [`shared-source-double-process-debug-demo.html`](../shared-source-double-process-debug-demo.html).
- [voice-fadein-clip-fadein-product.md](./voice-fadein-clip-fadein-product.md) — Audible amplitude dip on the new voice's fade-in side of a crossfade between regions with different `sourceUuid`s. Mechanism confirmed: `PitchVoice` multiplied its 20 ms voice-fade-in with the clip's gain buffer, turning a linear clip fade-in into a quadratic ramp (−1.20 dB at τ −10 ms). **Fixed in SDK 0.0.159 on both engines** (openDAW#312 closed; `PitchVoice` now combines the fades by `Math.min` — OPENDAW scans at −0.05 dB vs the −0.00 dB ALIGNED target). Repro: [`voice-fadein-clip-fadein-product-debug-demo.html`](../voice-fadein-clip-fadein-product-debug-demo.html) (live audition + offline scan; configuration uses two Tape tracks to avoid the deletion described in `project-copy-deletes-overlapping-regions.md`).

- [project-copy-deletes-overlapping-regions.md](./project-copy-deletes-overlapping-regions.md) — `project.copy()` silently deletes both regions when two `AudioRegionBox`es on the same track have overlapping timeline ranges. The live engine plays the overlap fine; the copy validator removes the regions and the offline-rendered buffer is all zeros (`queryLoadingComplete` returns true on iteration 0 because there's nothing to load). The deletion also fires from a more subtle path — `AudioRegionBox.position` is `Int32Field` (truncates on assignment) while `duration` is `Float32Field`, so a fractional PPQN value assigned to both creates a sub-PPQN overlap with no overlap intended by the consumer. Workaround we used: place each region on its own track; pick seam-in-seconds values whose PPQN is integer. Repro: voice-fade CROSSFADE configuration in [`voice-fadein-clip-fadein-product-debug-demo.html`](../voice-fadein-clip-fadein-product-debug-demo.html); contrast with the separate-track configuration in [`pure-webaudio-target-debug-demo.html`](../pure-webaudio-target-debug-demo.html) which renders normally.

- [seconds-overlap-validation-unit-mismatch.md](./seconds-overlap-validation-unit-mismatch.md) — `ProjectValidation.validate()` compares `right.position < left.position + left.duration` on raw box values; Seconds-timeBase regions store `duration` in seconds while `position` is PPQN, so the comparison mixes units and realistic Seconds overlaps survive `project.copy()` undetected. Both live probe paths (`Project.invalid()`, `RegionClipResolver.validateTrack`) skip Seconds tracks entirely. Musical overlaps are detected and deleted; Seconds overlaps are not. Pin: studio-core 0.0.152 / studio-adapters 0.0.116.

- [wasm-ensure-ready-second-context.md](./wasm-ensure-ready-second-context.md) — `WasmEngine.ensureReady(context)` registers the wasm processor module only on the FIRST context it is ever called with; for any later `BaseAudioContext` it returns `true` without registering, and booting the WASM `EngineWorklet` there throws `InvalidStateError: 'engine-wasm-processor' is not defined in AudioWorkletGlobalScope`. The deprecated-but-exported `AudioOfflineRenderer.start` is the public-API route into the bug; `OfflineEngineRenderer` (`variant: true`) is immune. Filed upstream as [openDAW#315](https://github.com/andremichelle/openDAW/issues/315) — **closed wontfix 2026-07-16**: use `OfflineEngineRenderer` for offline renders; "The Typescript audio-engine will be removed soon." Repro: [`wasm-ensure-ready-second-context-debug-demo.html`](../wasm-ensure-ready-second-context-debug-demo.html).

- [recording-finalize-no-terminal-state.md](./recording-finalize-no-terminal-state.md) — A recording's `SampleLoader` (the `RecordingWorklet`) only ever reaches `{type: "loaded"}`; a finalization that never completes leaves it in `{type: "record"}` with no `{type: "error"}` and no terminal event, so consumers can't tell a slow finalization from a failed one (a timeout is the only net). `#finalize()` is gated on `numberOfFrames >= limitSamples` and its rejections are swallowed by `.catch(console.warn)`. The debug page measures the gate margin directly — 26/26 valid cycles finalized with a thin ~32–44 ms surplus (`deficit` −1407…−1920 frames). The rare wild symptom (stuck loader, seen 2×) did not reproduce on demand this session; the page is retained as the capture instrument (`window.__finalizeDebug`). Pin: SDK 0.0.154 / studio-core 0.0.152. Repro: [`recording-finalize-debug-demo.html`](../recording-finalize-debug-demo.html).

**Target / reference comparison:**

- [`pure-webaudio-target-debug-demo.html`](../pure-webaudio-target-debug-demo.html) — Same crossfade scenario (two slightly phase-offset 440 Hz sines, 40 ms linear crossfade at the seam) rendered three ways for A/B/C comparison. **ALIGNED** (pure Web Audio + phase-correlation shift + linear crossfade) is the audible target — clean unity-sum through the crossfade. **UNALIGNED** (pure Web Audio, shift = 0) is the control showing what a phase-mismatched linear crossfade sounds like. **OPENDAW** runs the same configuration through OpenDAW's `TapeDeviceProcessor` with each region on its own Tape track (workaround for the `project.copy()` overlap-deletion bug). The audible delta between OPENDAW and ALIGNED is the gap the engine artifacts documented above currently produce — OPENDAW should match ALIGNED once those are resolved.
